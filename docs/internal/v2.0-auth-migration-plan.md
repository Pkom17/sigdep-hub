# SIGDEP-3 v2.0 — Plan de migration auth : Keycloak → Spring Security pur

Document de référence pour la migration de l'authentification. Rédigé
après l'échec du déploiement pilote v1.0.4 sur `sigdephub-v3.itech-civ.org`
où Keycloak a accumulé tant de friction (redirect URIs hardcodées,
realm import à patcher, base/user dédiée, SPA bouton « Se connecter »
inactif, etc.) qu'on a décidé de tout retirer.

## Pourquoi

- Keycloak est disproportionné pour SIGDEP-3 : pas de SSO, pas de
  federation LDAP, une seule app cliente.
- Le legacy `/Users/pascal/dev/sigdep/sigdepHub/` tournait depuis
  des années en Spring Security pur (BCrypt + JWT RS256) — preuve
  que c'est suffisant pour ce cas d'usage.
- Chaque problème rencontré pendant le pilote était spécifique à
  Keycloak. Spring Security élimine la classe entière.

## Décisions de design figées

| Sujet | Choix |
|-------|-------|
| Modèle de rôles | Hybride : 8 rôles existants (`SUPER_ADMIN`, `IT_ADMIN`, `NATIONAL_VIEWER`, `REGIONAL_COORD`, `DISTRICT_COORD`, `SITE_USER`, `ANALYST`, `AUDITOR`) + concept `UserLevel` repris du legacy |
| Algo signature JWT | **HS256** (HMAC-SHA256, clé symétrique en `.env`) |
| TTL JWT | Access 1h + refresh 7 jours |
| Scope geo dans JWT | Claims `regionId` / `districtId` / `siteId` inclus dans le JWT (compatible `AuthScope.java` existant) |
| Auth agent sigdep-sync | **API key opaque** (UUID) vérifiée en DB à chaque requête |
| Génération API key | Bouton « Générer une clé » sur `/app/sites/<code>` dans la console, affichée 1× seulement, hash BCrypt en DB |
| Seed SUPER_ADMIN initial | `SIGDEP_ADMIN_EMAIL` + `SIGDEP_ADMIN_PASSWORD` lus au 1er boot si la table `auth.users` est vide |
| Reset password | OTP 6 chiffres par email (repris du legacy) |
| Hash mot de passe | BCrypt |
| Refresh token | Stocké en DB (table `auth.refresh_tokens`), renvoyé en JSON (pas en cookie) |

## Architecture cible

```
┌──────────────┐
│    nginx     │ TLS, reverse-proxy
└──────┬───────┘
       │
   ┌───┴────────┬─────────────┐
   ▼            ▼             ▼
console-web  console-api  ingestion-api
(SPA)        ├─ /api/auth/*   ├─ /api/v1/sync/*  ← X-API-Key UUID
             ├─ /api/v1/users      (vérifié en DB)
             └─ /api/v1/*    (toutes les autres routes : JWT HS256)
             ↓                ↓
           ┌────────────────────┐
           │   PostgreSQL        │
           │   + auth.users      │
           │   + auth.refresh_tokens
           │   + auth.api_keys   │
           │   + auth.user_geo_scope (REGION/DISTRICT/SITE)
           │   + auth.password_reset_otp
           └────────────────────┘
```

Plus de Keycloak. Plus de base `keycloak`. Plus de realm-sigdep.json.
Plus de `react-oidc-context`. Plus de `KEYCLOAK_*` env vars.

## Schéma DB (Liquibase v2.0)

Tables sous schema `auth` :

### `auth.users`
- `id BIGSERIAL PK`
- `email VARCHAR(255) UNIQUE NOT NULL`
- `password_hash VARCHAR(255) NOT NULL` (BCrypt)
- `display_name VARCHAR(255) NOT NULL`
- `role VARCHAR(32) NOT NULL` (un des 8 rôles)
- `user_level VARCHAR(16) NOT NULL` (NATIONAL/REGION/DISTRICT/SITE)
- `active BOOLEAN NOT NULL DEFAULT TRUE`
- `password_expired BOOLEAN NOT NULL DEFAULT FALSE` (force change au prochain login)
- `last_login_at TIMESTAMP NULL`
- `created_at TIMESTAMP NOT NULL DEFAULT NOW()`
- `updated_at TIMESTAMP NOT NULL DEFAULT NOW()`

### `auth.user_geo_scope`
- `id BIGSERIAL PK`
- `user_id BIGINT NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE`
- `region_id BIGINT NULL REFERENCES core.regions(id)`
- `district_id BIGINT NULL REFERENCES core.districts(id)`
- `site_id BIGINT NULL REFERENCES core.sites(id)`
- contrainte : exactement un des 3 IDs non-null selon `user_level`

### `auth.refresh_tokens`
- `id UUID PK` (la valeur exposée au client)
- `user_id BIGINT NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE`
- `expires_at TIMESTAMP NOT NULL`
- `revoked BOOLEAN NOT NULL DEFAULT FALSE`
- `created_at TIMESTAMP NOT NULL DEFAULT NOW()`

### `auth.password_reset_otp`
- `id BIGSERIAL PK`
- `user_id BIGINT NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE`
- `otp_hash VARCHAR(255) NOT NULL` (BCrypt du code 6 chiffres)
- `expires_at TIMESTAMP NOT NULL` (15 min)
- `used_at TIMESTAMP NULL`
- `created_at TIMESTAMP NOT NULL DEFAULT NOW()`

### `auth.api_keys`
- `id BIGSERIAL PK`
- `site_id BIGINT NOT NULL REFERENCES core.sites(id) ON DELETE CASCADE UNIQUE`
- `key_hash VARCHAR(255) NOT NULL` (BCrypt du UUID, le UUID brut n'est jamais stocké)
- `key_prefix VARCHAR(8) NOT NULL` (8 premiers chars pour identifier rapidement la clé sans déchiffrer)
- `created_by BIGINT NOT NULL REFERENCES auth.users(id)`
- `created_at TIMESTAMP NOT NULL DEFAULT NOW()`
- `last_used_at TIMESTAMP NULL`
- `revoked_at TIMESTAMP NULL`

## Plan d'exécution (4 PRs)

### PR1 — Fondations backend (1/2 jour)
1. Liquibase changesets 035-038 (tables auth.*)
2. Entités JPA : `User`, `UserGeoScope`, `RefreshToken`, `ApiKey`, `PasswordResetOtp`
3. `UserDetailsService` + `UserPrincipal` impl Spring Security
4. `JwtService` (sign/verify HS256, claims = sub, role, regionId, districtId, siteId)
5. `JwtAuthFilter` (lit `Authorization: Bearer ...`)
6. `ApiKeyAuthFilter` (lit `X-API-Key: <uuid>`)
7. `SecurityConfig` console-api : stateless, CORS, `/api/auth/login` public, reste authentifié
8. `SecurityConfig` ingestion-api : ApiKeyAuthFilter sur `/api/v1/sync/**`
9. `AuthController` : POST `/api/auth/login`, GET `/api/auth/me`, POST `/api/auth/refresh`, POST `/api/auth/logout`
10. Seed admin au boot via `SIGDEP_ADMIN_EMAIL`/`SIGDEP_ADMIN_PASSWORD`
11. **Test manuel** : `curl /api/auth/login` retourne un JWT, `curl /api/v1/sites` avec le JWT marche.

### PR2 — SPA login (1/4 jour)
1. Retirer `react-oidc-context`, `oidc-client-ts` du `package.json`
2. Nouvelle page `Login.tsx` (formulaire email+password, POST `/api/auth/login`)
3. Nouveau hook `useAuth` qui lit le JWT du localStorage (clé `sigdep.access_token`, `sigdep.refresh_token`)
4. `apiClient` ajoute `Authorization: Bearer ${accessToken}` à chaque requête
5. `ProtectedRoute` redirige vers `/login` si pas de JWT
6. Intercepteur 401 → tentative refresh → si échec, logout + redirect login
7. **Test manuel** : login dans le navigateur, naviguer dans la console.

### PR3 — Gestion users + agent (1/2 jour)
1. Refondre `UsersController` (CRUD complet, plus de KeycloakAdminService)
2. Page React `Utilisateurs` adaptée au nouveau backend (créer, lister, désactiver, reset password manuel ou OTP, assigner rôle+scope)
3. Page React `Sites` : bouton « Générer une clé API » par site, affichage 1× du UUID
4. Agent `sigdep-sync` : remplacer `OAuthAuthenticator` par `ApiKeyAuthenticator` (header `X-API-Key`), retirer `SIGDEP_KEYCLOAK_*` env vars, ajouter `SIGDEP_API_KEY`
5. Tester un cycle de sync agent → hub avec la nouvelle auth.

### PR4 — Retirer Keycloak + docs + release (1/4 jour)

2. Retirer `infra/keycloak/`, `KC_*` env vars, `KEYCLOAK_ADMIN_CLIENT_SECRET`

## Risques identifiés

- **Migration des données** : aucune, on part de zéro pour le pilote. Pas de users à migrer depuis Keycloak (il n'y avait que l'admin de test).
- **Casser les tests** : pas de tests automatisés (dette technique reconnue), donc rien à casser.
- **Sync des sites existants** : pas de sites en prod encore, donc la rotation des API keys ne pose pas de problème.
